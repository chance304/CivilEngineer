## Phase 2: MEP Engine

### What Was Built

A fully deterministic MEP (Mechanical, Electrical, Plumbing) routing engine that runs
as a LangGraph node between geometry generation and engineer review.

**Key design decisions:**
- All routing is deterministic — no LLM involvement. The LLM is strictly excluded from
  numeric decisions (consistent with Critical Rule #1).
- Electrical conduit routing uses A* pathfinding on a 0.25m-resolution grid derived from
  wall segment obstacles.
- Plumbing stacking is deterministic: wet rooms within 1.5m horizontal offset across floors
  share a vertical stack.
- NBC rule enforced: no branch plumbing pipes routed through bedroom room bounds.
- Panel phase selection: 1-phase for total load ≤ 10 kVA, 3-phase otherwise.

### Bug Fixed During Implementation

**`NameError: name 'extract_high_load_appliances' is not defined`** on module import of `questions.py`.

Cause: The 6 MEP extractor functions were appended at the bottom of the file, but the
`QUESTIONS` list (a module-level list literal defined earlier) referenced them directly
in `extractor=` fields. Python evaluates module-level literals at import time, so functions
must be defined before the list.

Fix: Moved the MEP extractor function block to immediately before the `QUESTIONS` list.

## Phase 1: Frontend — Next.js 14 Portal

### What Was Built

A complete multi-page portal. All 39 TypeScript files written; no pages are stubs.

**Tech stack:**
- Next.js 14 App Router (TypeScript, strict mode)
- Tailwind CSS with shadcn/ui CSS variable tokens
- TanStack Query v5 for server state
- Zustand (persisted) for client state
- React Hook Form + Zod for all forms
- Sonner for toast notifications
- Vitest for component tests

**API proxy:** `next.config.js` rewrites `/api/v1/*` → `http://localhost:8000/api/v1/*`

### Pages (17 total)

| Route | Description |
|-------|-------------|
| `/login` | Email + password login, stores JWT in Zustand + httpOnly refresh cookie |
| `/forgot-password` | Email input for password reset |
| `/dashboard` | Project list with status badges, empty state CTA |
| `/projects/new` | 3-step wizard: Basic Info → Jurisdiction (with setback preview) → Options |
| `/projects/[id]/plot` | DWG/DXF drag-and-drop upload, polygon preview, setback overlay |
| `/projects/[id]/interview` | Chat-bubble interview UI with progress counter |
| `/projects/[id]/design/[sessionId]` | 8-step real-time progress timeline via WebSocket |
| `/projects/[id]/design/[sessionId]/review` | Split-panel: floor plan viewer + approve/revise/abort |
| `/projects/[id]/files` | Session list with typed download links (DXF/PDF/IFC/DWG/MEP) |
| `/admin/llm-config` | LLM provider, model, API key, temperature, test connection |
| `/admin/building-codes` | PDF upload, status badges, extracted rule review, activate |
| `/admin/users` | User list with role badges, invite modal |
| `/settings` | Display name, read-only email/role, password change |

Plus nested layouts: `(auth)/layout.tsx` (minimal), `(portal)/layout.tsx` (sidebar + header + JWT guard).

### Key Components

**Viewers (all SVG, no third-party rendering lib):**
- `FloorPlanViewer.tsx` — multi-floor tabs, rooms with type-colour fill, walls, columns, doors, windows, zoom+pan via CSS transform, compliance badges
- `ElevationViewer.tsx` — front/rear/left/right tabs, floor bands, openings at correct heights
- `Building3DViewer.tsx` — isometric projection from vertices+edges JSON, ±45° rotation

**`lib/dxf-renderer.ts`** — pure functions that convert FloorPlan JSON → SVG element props. No DXF parser required; data comes from the API as structured JSON.

**`lib/websocket.ts`** — singleton WebSocketManager with 3s auto-reconnect, typed `on(event, handler)` → returns unsubscribe function, JWT injected in URL on connect.

**`store/useAppStore.ts`** — Zustand store with `partialize` to persist only auth state (user, accessToken, firm). jobProgress and approvalRequest are in-memory only.
