## Item 1: ZIP Package Download

**Problem:** Output files were only written to the Celery worker's local filesystem.
The `/files` endpoint returned an empty array. There was no way to download the
complete set of drawings.

**Solution:**

### Backend (`jobs/design_job.py`)
- Added `_upload_output_files()` — reads each output file from disk after the pipeline
  completes, uploads to S3 (`civilengineer-projects` bucket, key prefix
  `outputs/{session_id}/{filename}`).
- Added `_classify_output_file(name)` — maps filename patterns to type strings
  (`dxf_floor_plan`, `dxf_elevation`, `dxf_3d`, `dxf_mep`, `pdf`, `ifc`, `dwg`).
- Job result now stores `output_file_records` (list of `{name, type, s3_key, size_bytes}`)
  instead of raw local paths.

### Backend (`api/routers/design.py`)
- `GET /projects/{id}/design/{sessionId}/files` — reads `output_file_records` from
  job result, generates fresh 1-hour presigned download URLs per file.
- `GET /projects/{id}/design/{sessionId}/files/zip` — downloads each file from S3 into
  an in-memory `io.BytesIO` ZIP archive, returned as `application/zip` with
  `Content-Disposition: attachment`.

### Frontend (`lib/api.ts`, `files/page.tsx`)
- `designApi.downloadZip()` — fetch with JWT header → blob → object URL → programmatic
  `<a>` click → cleanup. Works around same-origin JWT constraint.
- `files/page.tsx` fully rewritten: per-session `SessionCard` components with lazy
  `SessionFilesPanel` (only fetches files when expanded), "Download All" ZIP button.

## Item 2: Documentation Completeness Gate + `finalized` Status

**Purpose:** Give engineers a formal checkpoint to assert that all required drawings
are present before handing off to client. Prevents accidental handoffs with missing
floor plans.

### Backend
- `JobStatus.FINALIZED` added to the enum.
- `finalized_at: datetime | None` column added to `DesignJobModel`.
- Migration `20260224_003_finalized_status.py` — adds `finalized_at` column.
- `POST /projects/{id}/design/{sessionId}/finalize`:
  - Returns 200 immediately if already finalized.
  - Returns 422 if session is not yet `completed`.
  - Checks required files (`dxf_floor_plan`, `pdf`, `site_plan.dxf`); 422 with full
    report if any are missing.
  - Advisory items (`dxf_elevation`, `dxf_3d`, `dxf_mep`) reported but don't block.
  - On success: sets `status = "finalized"`, `finalized_at = now()`.

### Frontend (`files/page.tsx`)
- "Finalize" button (emerald) shown only on `completed` sessions.
- After finalize attempt: completeness report panel rendered (green dot = present,
  red dot = missing; yellow `AlertCircle` for advisory).
- Finalized sessions get emerald border + `ShieldCheck` icon.
- `designApi.finalize()` typed with full completeness response.

## Item 3: Flooring & Finishes Schema + Interview Questions

### `schemas/design.py`
New types added:

```python
class FloorFinish(StrEnum):
    CONCRETE = "concrete"   # plain screed — cheapest
    VINYL    = "vinyl"
    TILE     = "tile"       # baseline (1.0×)
    MOSAIC   = "mosaic"
    GRANITE  = "granite"
    HARDWOOD = "hardwood"
    MARBLE   = "marble"     # most expensive (2.8×)

class FinishSpec(BaseModel):
    flooring:   FloorFinish = FloorFinish.TILE
    wall_paint: str = "standard"   # standard | premium | texture
    ceiling:    str = "plaster"    # plaster | pop | false_ceiling | wood_panel
```

`DesignRequirements` gains:
```python
finish_overrides: dict[str, FinishSpec] = Field(default_factory=dict)
```
Keys are `RoomType` string values (e.g. `"bedroom"`, `"bathroom"`).

### `requirements_interview/questions.py`
Three deterministic extractor functions:
- `extract_floor_finish(text)` — keyword matching → `FloorFinish` enum value
- `extract_ceiling_finish(text)` — returns `"wood_panel"`, `"false_ceiling"`, `"pop"`, or `"plaster"`
- `extract_wall_paint(text)` — returns `"texture"`, `"premium"`, or `"standard"`

Four new interview questions (phase `"finishes"`, all optional):
- `floor_finish_dry` — bedrooms / living / dining / home_office / pooja_room
- `floor_finish_wet` — bathroom / toilet / kitchen
- `ceiling_finish` — applied to all rooms in both groups
- `wall_paint` — applied to dry rooms

`answers_to_requirements()` updated to populate `finish_overrides` dict.

## Item 4: Cost Estimator — Finish Overrides + Tier Comparison

### `output_layer/cost_estimator.py`

**Finish multiplier tables** (applied to the room's base finish rate):

| Multiplier | Standard value | Range |
|---|---|---|
| `_FLOORING_MULT[FloorFinish.TILE]` | 1.00 (baseline) | 0.50 (concrete) → 2.80 (marble) |
| `_WALL_PAINT_MULT["standard"]` | 1.00 | up to 1.60 (texture) |
| `_CEILING_MULT["plaster"]` | 1.00 | up to 2.50 (wood_panel) |

Weighted composite: `0.50 × floor_mult + 0.30 × wall_mult + 0.20 × ceil_mult`

**`CostEstimator` changes:**
- Constructor now accepts `finish_overrides: dict[str, FinishSpec] | None`
- Per-room: looks up `finish_overrides[room_type]`, computes multiplier, scales finish cost
- Structure cost is always unaffected by finish overrides
- `_tier_grand_total(building, grade)` helper — re-runs the estimate at a given grade
  with no overrides, used to populate `tier_comparison`

**Schema changes:**
- `RoomCost.flooring_used: str | None` — set when override applies
- `CostEstimate.tier_comparison: dict[str, float]` — `{"basic": ₹X, "standard": ₹Y, "premium": ₹Z}`

### `output_layer/pdf_exporter.py` (`_cost_page`)
Two new sections appended after the room-type breakdown table:
1. **Flooring Selections Applied** — table showing only rooms with an override
   (floor / room type / material / finish cost)
2. **Grade Comparison** — three-row table (basic / standard / premium) with
   the selected grade highlighted, sourced from `tier_comparison`

### `agent/nodes/draw_node.py`
Parses `requirements["finish_overrides"]` (raw dicts) into `dict[str, FinishSpec]`
and passes to `CostEstimator(finish_overrides=...)`. Malformed overrides silently
skipped so pipeline never crashes on bad input.

## Item 5: Client Approval Interface (Viewer Role)

**Purpose:** After an engineer finalizes a session, the client (viewer role) can
formally approve the design or request changes — separate from the mid-pipeline
engineer approval interrupt.

### Backend — `db/models.py`
New `ClientApprovalModel` table:

| Column | Type | Notes |
|---|---|---|
| `approval_id` | UUID (PK) | auto-generated |
| `project_id` | FK → projects | |
| `session_id` | str, indexed | |
| `firm_id` | FK → firms | |
| `submitted_by` | FK → users | viewer's user_id |
| `submitted_at` | datetime | |
| `action` | str | `"approved"` \| `"revision_requested"` |
| `notes` | str | required when revision_requested |

### Backend — Migration
`20260224_004_client_approvals.py` — creates table + 3 indexes.

### Backend — `api/routers/design.py`
Two new endpoints (both gated on `Permission.DESIGN_READ` so viewer role can call them):

**`GET /projects/{id}/design/{sessionId}/client-approval`**
- Returns most recent approval record, or `{has_approval: false}` if none.

**`POST /projects/{id}/design/{sessionId}/client-approve`**
- Validates: `action` ∈ `{"approved", "revision_requested"}`
- `notes` required (non-whitespace) when `action = "revision_requested"`
- Session must be in `completed` or `finalized` status
- Creates new `ClientApprovalModel` row (multiple submissions allowed, GET returns latest)
- Returns the new approval record

### Frontend — `lib/api.ts`
- `designApi.getJob(projectId, sessionId)` — GET single job with `result` payload
- `designApi.getClientApproval(projectId, sessionId)` — GET current client sign-off
- `designApi.clientApprove(projectId, sessionId, action, notes?)` — POST decision

### Frontend — `projects/[id]/design/[sessionId]/client-review/page.tsx`
